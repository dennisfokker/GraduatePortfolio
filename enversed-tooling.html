ENVERSED TOOLING

Of course Enversed didn’t just make a game and put it out there, certain tooling was used whilst developing on projects. A list and description can be found below.

Management

Jira was used for managing all user stories, their subtasks and potential bugs. These tasks were tracked using a Kanban board using different swimming lanes and priorities.

Once a user story is accepted and subtasks are made, the subtasks will go past the following swimming lanes: To-Do > Preparation <> Preparation review > Work <> Testing <> Work review > Done.

For further information about how Jira was used, see Enversed workflow.

Source control

Git is used to manage source control. The master branch is reserved for (tagged) releases while the develop branch is as the name suggests.

At first feature-branches were used, the develop branch only containing completed subtasks and each subtask creating a new branch on which could be worked on before merging back into develop.

Later on, Git LFS was integrated into the workflow. Blueprints are binary files, which can’t be merged. Previously we simply had to communicate who was working with what file. With Git LFS storage is better handled (it occupies less space on the harddrive), as does it have a feature for locks. Locks can be setup on a file to indicate you are working on it, and prevents and informs other developers about this.

In order to optimally integrate this, a plugin was also used in Unreal Engine, which supports LFS locks with icons on the files and warnings when working with (un)locked files.

Communication

Team communication is performed over Slack. A few general channels are there (such as a team or discuss-unreal channel) as well as a channel for each project.

The idea is that (almost) all communication should go over Slack, that way you won’t bother people directly when they might not have any time, or a third person might chime in who has the correct answer to the question.

Only once a conversation has been initiated on Slack can it potentially be changed to an in-person discussion. It’s important, however, to summarise every discussion, as this makes it easier for other people to know what to do when they have similar questions.

Documentation

Documentation is put in Confluence. It’s important to categorise each “page” in the correct project and (potential) sub-category. Each project should always have a readme-esque page to describe how to start working on it, what client feedback there is and other notes.

Whenever a big task is picked up, this should also receive a page on Confluence, as this makes it much easier to reference at a later date.
