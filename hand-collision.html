HAND COLLISION

A system used to make the visual representation of your hand physical, whilst minimizing the chance of breaking the immersion of the game.
It provides extra visualisation for your real life hand position, which only becomes visible once your hand starts making illegal movements, like going into a wall.

CONCEPT

Every system comes with their own requirements and user story. While the user story is something predetermined by the project lead, the requirements are something that comes as the team discusses the best method to tackle the problem.

During concepting it's important to keep in mind what the player or developer might (not) be expecting; their hand shouldn't explode nor should their screen go black whenever hitting a wall.

REQUIREMENTS

Needs to be clear to the player what is happening;
Player should not be able to interact with items on opposite side of immovable object;
Players still need to be able to interact with items (when not blocked by immovable objects).

Collision

Something already mentioned earlier is that the system uses tracing, instead of using “normal” collision. Specifically “sphere traces by channel” are used: a sphere simple shape is used to continuously test from point A to B, colliding only which objects that block the specified channel. This provides more freedom than using ordinary physics collision, without sacrificing efficiency.

The system checks for two criteria, in order. If at least one of these succeed, your character’s hand will move to the position of the respective motion controller.

  1. Check if there’s anything between your latest character’s hand position and the position of the motion controller.
  2. Check if there’s anything between your camera’s position and the position of the motion controller.

The reason behind this order is because the distance of check 1 is generally* much shorter than that of 2, resulting in cheaper traces.

Configurability

The system contains 3 settings: whether it’s active, hand size and visibility threshold.

Whether it’s active

This is simple enough. Should the system actually be running, or do we currently not care if your character’s hand is inside the wall?

This can be used when the project has no need for this system, or when certain actions wouldn’t work nicely and need the system temporarily disabled.

An example of a system that doesn’t work along nicely is the climbing system, due to the order in which your body is moved and the system performs its check.

Hand size

Depending on the visualisation your hand size can differ, either on project basis or based on the current state of your character. The traces use this size.

This setting is visualised with a sphere simple collision shape without any collision enabled. This visualisation is only visible in the viewport (engine-time) and make it easier to see scale, compared to a float (radius in cm).

Visibility threshold

When the hand collides with something and stops moving, the “ghost hand” doesn’t instantly become visible. Instead a distance threshold can be chosen when the ghost should pop up.

This prevents clipping, when the ghost hand and character hand are at nearly the same place.

Clarity

An important thing about clarity, whilst maintaining immersion, is to be noninvasive. Examples would be: don’t make essential things disappear, try to be predictable and, of curse, don’t scare the player.

Currently, when the character’s hand “collides”, all it does is stop moving. Nothing too weird. And given the system is properly set up (see configurability), should also be somewhat expected behaviour.

Aside from stopping the character’s hand, it also displays a “ghost hand”. This ghost could be either the same mesh (but with different texture) as that of the character, or use a mesh resembling the real-life motion controller used.

This mesh is editable on a per-project basis (and if you really want to can also be changed during run-time (like any mesh component). Using a character’s hand mesh should help keeping immersion whilst using a controller mesh should provide more clarity that the thing that’s moving is not “part of the game”, but instead some form of indication, like the room-scale boundaries that pop up as you reach the edges of your setup.
